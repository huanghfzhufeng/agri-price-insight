# 开发文档

## 1. 文档目标

本文档用于把当前仓库从“可运行骨架”推进到“完整可落地的优秀毕业设计项目”。

本项目最终目标不是只做一个演示页面，而是交付一套可以支撑论文、答辩、源码提交和系统演示的完整系统：

- 有稳定的数据来源
- 有可解释的数据处理流程
- 有前后端分层架构
- 有统计分析与预测能力
- 有测试文档和测试结果
- 有清晰的开发过程记录

## 2. 项目最终定位

建议最终题目统一为：

《基于 Python 的农产品价格数据分析、预测与可视化系统的设计与实现》

最终系统定位：

- 面向农产品价格监测与辅助分析场景
- 支持价格数据采集、清洗、入库、查询、统计分析、预测预警和可视化展示
- 支持管理员对数据源、任务日志和系统运行状态进行管理

## 3. 当前基线

当前仓库已经完成以下基础能力：

- 已完成前后端工程初始化
- 已完成 FastAPI 后端骨架
- 已完成 React 前端骨架
- 已完成系统概览、数据查询、预测预警三个原型页面
- 已完成 SQLite 演示数据库与种子数据
- 已完成 GitHub 仓库发布

这意味着后续工作不再是从零开始，而是在现有骨架上逐步做“真实化”和“系统化”。

## 4. 总体开发原则

- 优先做“完整闭环”，再做“高级算法”
- 优先保证数据链路稳定，再优化模型效果
- 优先实现论文可写、答辩可演示的功能
- 每完成一个需求，更新 [TODO.md](/Users/zeke/agent%20dome/PYBS/TODO.md) 勾选状态
- 当前按你的要求只维护 `main` 分支

关于 PR：

- GitHub 标准 Pull Request 是基于“比较分支”工作的
- 如果始终只在 `main` 上开发，则无法进行标准的“功能分支 -> PR -> 合并到 main”流程
- 因此当前执行策略调整为：`一个需求 = 一个独立提交 = 直接推送 main`
- 如果你后续允许短期功能分支，再恢复标准 PR 流程

参考：

- GitHub Docs: [About pull requests](https://docs.github.com/en/pull-requests/collaborating-with-pull-requests/proposing-changes-with-pull-requests/about-pull-requests)

## 5. 数据获取完整方案

### 5.1 数据源设计原则

毕业设计的数据源必须满足：

- 来源权威
- 可持续获取
- 字段足够支撑分析
- 能解释清楚采集方式

### 5.2 主数据源

建议采用“1 主 2 辅”的数据策略。

#### 主数据源 A：农业农村部数据查询平台

用途：

- 作为系统核心价格数据来源
- 支撑历史价格趋势和价格指数分析

官方入口：

- [农业农村部数据查询](https://data.moa.gov.cn/)

建议采集内容：

- 农产品批发价格 200 指数
- 重点农产品价格监测数据
- 日期维度价格数据

#### 主数据源 B：农业农村部每日批发市场价格简报

用途：

- 补充当天监测摘要
- 用于构造“每日更新数据”与“涨跌提醒”

官方示例：

- [2025年2月11日农产品批发价格简报](https://www.moa.gov.cn/xw/zxfb/202502/t20250211_6469865.htm)

该类官方简报明确说明其日常监测覆盖重点品种，可作为论文中“数据权威性”依据。

### 5.3 辅助数据源

#### 辅助数据源 C：农业农村部农产品供需形势分析月报

用途：

- 为论文中的市场背景分析和预测解释提供支撑
- 可提取月度走势判断、供需变化和风险因素

官方示例：

- [2025年3月农产品供需形势分析月报](https://www.moa.gov.cn/ztzl/nybrl/rlxx/202503/P020250324376108713746.pdf)

#### 辅助数据源 D：国家统计局农产品价格调查制度说明

用途：

- 用于论文中的“数据背景与调查口径”描述
- 强化系统研究背景的严谨性

官方说明：

- [主要农产品及农用生产资料价格调查制度的主要内容](https://www.stats.gov.cn/fw/bmdcxmsp/bmzd/202303/t20230330_1938225.html)

### 5.4 第一阶段采集范围

为了保证项目可落地，第一阶段不追求“大而全”，建议只采集以下重点品种：

- 猪肉
- 鸡蛋
- 大蒜
- 大白菜
- 富士苹果
- 大豆
- 玉米
- 西红柿

建议先完成 `6-8` 个品种，后续再扩展到更多品类。

首批必须入库的标准字段：

- `stat_date`

- `category`
- `market_name`
- `region`
- `min_price`
- `max_price`
- `avg_price`
- `unit`
- `source`
- `crawl_time`

扩展字段：

- `holiday_factor`

### 5.6 数据采集实现方案

#### 阶段 1：演示型稳定采集

目标：

- 先完成从官方页面抓取文本或表格并入库
- 保证系统可持续更新

实现：

- 编写 `backend/scripts/fetch_moa_daily.py`
- 编写 `backend/scripts/normalize_prices.py`
- 将抓取结果先写入原始表
- 再清洗后写入标准价格表

#### 阶段 2：定时任务化

目标：

- 实现每日自动采集
- 保留任务日志和失败重试

实现：

- 集成 APScheduler
- 增加 `task_log` 表
- 增加管理端任务状态查看

#### 阶段 3：分析数据集构建

目标：

- 构建适合统计分析和预测建模的宽表或特征表

实现：

- 生成日级宽表
- 生成周级聚合表
- 生成月级聚合表
- 生成模型训练特征表

### 5.7 数据清洗规则

必须落地以下规则：

- 去重：同品种、同市场、同日期数据去重
- 格式标准化：日期统一为 `YYYY-MM-DD`
- 单位标准化：斤、公斤、吨分别统一
- 空值处理：缺失值标记，不直接删除原始记录
- 异常值处理：IQR 或 Z-Score 标记
- 名称映射：同义品名统一到标准品类

### 5.8 数据质量要求

毕业设计答辩中必须能说清楚以下质量控制：

- 数据完整率
- 异常值比例
- 清洗后保留率
- 采集成功率
- 接口响应耗时

## 6. 后端开发方案

### 6.1 后端目标

后端最终要从“演示接口”升级为“业务接口 + 数据任务 + 分析接口”。

### 6.2 功能模块拆分

后端模块最终分为：

- 系统模块：健康检查、配置、基础字典
- 认证模块：登录、角色、鉴权
- 数据采集模块：采集任务、重试、日志
- 数据查询模块：条件筛选、分页、导出
- 统计分析模块：同比、环比、趋势、波动率
- 预测预警模块：模型训练、预测结果、阈值预警
- 系统管理模块：数据源配置、任务日志、报表导出

### 6.3 数据库升级路线

当前：

- SQLite 用于演示

目标：

- MySQL 作为正式毕业设计数据库

迁移顺序：

1. 保留 SQLite 本地开发能力
2. 补充 MySQL 配置
3. 完成建表迁移脚本
4. 完成开发环境切换

### 6.4 分析与模型路线

必须完成：

- 日均价、周均价、月均价
- 环比与同比
- 最大值、最小值、波动率
- Top 涨跌排行
- 区域对比分析

建议完成：

- 移动平均
- 季节性分析
- 异常值识别

预测模型建议：

- 第一版：移动平均或线性趋势基线
- 第二版：Prophet
- 第三版：RandomForestRegressor 或 XGBoost 做对比

论文中重点写：

- 为什么选这些模型
- 模型评价指标为什么采用 `MAE / RMSE / MAPE`
- 哪个模型更适合农产品价格预测
